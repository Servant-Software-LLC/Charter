STANDING USER PREFERENCES / DIRECTIVES (David):
- EVERY deferral gets a GitHub issue ("if we just state it in the plan, it will be
  forgotten after the plan is executed"). Distinction he drew: DEFERRAL ("maybe later")
  -> issue; DECISION ("we chose not to", e.g. telemetry-off) -> NO issue.
- Zero telemetry is a hard preference ("Yes, zero telemetry is my preference").
- The autonomy DIAL is David's idea: auto-proceed vs halt-and-escalate is an
  adjustable dial; AI assesses criticality and the dial arbitrates. But HARD BLOCKERS
  override the dial ("even if the dial is set to not stop at all costs, it will have
  to stop").
- Discoverability meta-rule: a shipped capability that requires grepping a 2,100-line
  skill to discover is itself a defect; a skill must surface its capabilities "at the
  altitude a consumer first looks" (-> Guardrails #354).
- Wave labels: never strip the wave-NN-slug prefix as a standalone change — that hides
  waves further; only remove it coupled to the subgraph work that replaces it.

KEY DECISIONS & RATIONALE (roads taken AND rejected):
- Charter did NOT start as markdown+directives. First sketch was "go totally MDX,
  keep my own OSS" (full MDX + Preact/React SSR + block components). markdown+`:::`
  directives (Markdig) won AFTER a format-research pass: gets ~90% of MDX's benefit
  without dragging a JS compiler+framework into a codebase with neither. The real
  driver is LEANNESS, not just "MDX can't run in C#".
- Load-bearing insight that de-risked the whole thing: commenting depends on the
  loopback SERVER, not the authoring FORMAT. "server vs static" and "HTML vs MDX vs
  markdown" are INDEPENDENT axes; annotations anchor to DOM, which doesn't care whether
  a human, agent, or compiler produced the HTML. So: render markdown -> HTML at BUILD
  time -> serve through the same server. Client-side/in-browser render was explicitly
  rejected (makes the HTML inert without a bundle; the exact property Lavish avoids).
- No standalone M0 spike: David argued the DAG makes a slice's orientation value
  redundant; assistant sharpened it ("a DAG is a specification, not evidence") and
  folded M0's risk-proofs (anchor-survival, handoff-validates) into wave-1 acceptance
  guardrails.
- Distribution = C# native binary + Homebrew (Guardrails #353), chosen over npx-TS and
  over a Go/Rust rewrite. Charter is greenfield and COULD have gone npx (like
  lavish/gnhf/firstmate) but deliberately matched the C# suite for consistency.
- Charter's CI/CD is a PORT of Guardrails' pipeline (already-debugged ci.3), which is
  why Charter's first release dry-run went green immediately.
- Charter.Core is intentionally ONE implementation task (not split model/renderer/
  source-map) to avoid Guardrails' #176 transitive-compile diamond (test project
  compiles as a unit, references all three types). A documented WEAK sizing, on purpose.

FIRSTMATE / GUARDRAILS / AUTONOMY:
- firstmate = agent-distribution system dispatching autonomous crewmates in isolated
  worktrees, escalating only real decisions; a tool it drives can't block on an
  interactive human loop. Two-tier skills: public skills/ (self-contained) vs private
  .agents/skills/ (metadata.internal:true). firstmate is Shell; gnhf is TypeScript.
- Autonomy dial three-layer model (Guardrails #361): (1) judgment call -> dial
  arbitrates halt-vs-best-guess by criticality/confidence; (2) transient hard blocker
  -> bounded wait + backoff with configurable ceiling; (3) permanent blocker or retries
  exhausted -> unconditional halt-and-escalate with full context. "The dial governs
  judgment, not physics." Transient-vs-permanent is itself a recorded judgment (why
  retryable + attempts burned).
- Charter #7 is deliberately THIN: its one decision is render-and-persist vs serve, no
  mid-run gates -> no dial. Autonomous Charter = headless render + persist deliverable
  to disk + exit, WITHOUT the loopback comment server / long-poll. The "why": firstmate
  has no human present, but we still want deliverables persisted for forensic hindsight.
- Durable intent beyond v1: a cross-orchestrator MDX-planning capability usable from
  gnhf + firstmate public tiers AND at Tricentis.

SECURITY & TRUST (re-derived from Lavish's source, not inherited):
- Store-concurrency risk origin: Lavish does whole-file read-modify-write of session
  JSON, so concurrent poll + prompts can race -> Charter's open item: single-writer/lock.
- Export MUST strip the annotation SDK (a static page can't route comments home). Lavish
  reverted PR #16 (asset injection) to protect "the HTML is the artifact" portability
  (Lavish issue #37). Charter's markdown/block front-end must be ADDITIVE; portable HTML
  stays canonical.
- CRLF->LF gotcha: a CRLF shebang breaks install.sh on macOS/Linux ("bad interpreter");
  pinned LF in .gitattributes (Charter's install.sh is a port -> inherits this).

CI/CD & TOOLING GOTCHAS (from porting Guardrails' pipeline):
- .slnx incompatibility: .NET 10 SDK emits .slnx that the 8.x SDK/CI can't read. Charter
  uses a classic .sln pinned to .NET 8.x via global.json. Don't regenerate the solution
  with a newer SDK.
- Release tag uses `workflow_run` trigger, NOT `release: published` (a GITHUB_TOKEN-
  created release does not fire event chains). `-ci.` tag guard skips NuGet publish so
  throwaway v0.0.0-ci.N dry-run tags don't pollute NuGet.
- TAP_PAT secret is per-repo; must be set org-wide (Servant-Software-LLC / Guardrails,
  Charter) or the Homebrew tap-bump fails.
- Sandbox static-analysis guard blocks `Remove-Item -Recurse -Force` when authoring
  guardrail scripts in this harness.
- macOS: a CLI binary can't be stapled (stapler only does .app/.dmg/.pkg); notarization
  is server-side, Gatekeeper checks online on first run; JIT/unsigned-exec entitlements
  may need tuning. Relevant when Charter's signed release happens.

PRIOR ART (opinions worth keeping):
- Lavish's entire layout-audit subsystem (overflow/clip/overlap + gate) exists ONLY
  because agents hand-write HTML — a cost Charter's vetted block components remove "by
  construction". Direct motivation for the block approach.
- visual-plan is MORE of a live webhost than Lavish, not less: its default deliverable
  is always a hosted Agent-Native Plan via plan.agent-native.com MCP; even local mode
  renders through the hosted UI. Corrects the intuition that visual-plan is the
  lighter/local option.
- Lavish tagline "HTML is the new markdown" -> pitching a markdown front-end as
  "markdown replaces HTML" collides head-on; hence Charter is framed independent/additive.
- Lavish telemetry = ~194 LOC best-effort HTTP POST to a hardcoded Umami endpoint,
  default-ON. The sins to avoid are vendor-SDK + hardcoded endpoint + default-on — a
  configurable HttpClient POST (or OTLP) has none. Sharpens Charter's telemetry rule (#6).
- Builder.io pipeline internals: MDX -> typed components -> normalized to JSON,
  validated with Zod -> renderer. Load-bearing part is the strict SCHEMA
  ("type-safety for natural language"), reproduced by markdown+directives-validated-to-
  C#-records minus the JS runtime.
- visual-recap premise ("the same plan data model serves both directions") is exactly
  Charter's direction-agnostic architecture -> keep plan-only assumptions out of the
  block model + source-map so recap-mode (#1) stays cheap.
